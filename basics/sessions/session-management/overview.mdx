Session management in Agno gives you control over how sessions are identified, named, and cached for optimal performance.

## Core Concepts

### Session IDs

Every session has a unique identifier (`session_id`) that tracks conversations across multiple runs:

- **Auto-generated**: If not provided, Agno generates a UUID automatically
- **Manual**: You can provide your own session IDs for custom tracking
- **Sticky**: Once set on an agent/team instance, the session_id persists across runs
- **Per-user**: Combine with `user_id` to track multiple users' sessions

### Session Names

Session names are human-readable labels that make it easier to identify and manage conversations:

- **Custom Names**: Set descriptive names manually
- **Auto-generated Names**: Let the AI generate meaningful names from conversation content
- **Searchable**: Use names to find sessions in your application UI
- **Updateable**: Rename sessions at any time

### Session Caching

Session caching stores the session object in memory to improve performance:

- **Faster Access**: Avoids repeated database lookups
- **Production Ready**: Ideal for high-throughput applications
- **Automatic**: Once enabled, caching is handled transparently
- **Configurable**: Enable per agent/team instance

## Basic Examples

### Custom Session ID

```python
from agno.agent import Agent
from agno.models.openai import OpenAIChat
from agno.db.sqlite import SqliteDb

agent = Agent(
    model=OpenAIChat(id="gpt-4o"),
    db=SqliteDb(db_file="tmp/agent.db"),
)

# Use your own session ID
agent.run("Hello", session_id="user_123_session_456")
```

### Session Naming

```python
# Set a custom name
agent.set_session_name(session_name="Product Discussion")

# Or auto-generate from conversation
agent.set_session_name(autogenerate=True)

# Retrieve the name
name = agent.get_session_name()
```

### Session Caching

```python
agent = Agent(
    model=OpenAIChat(id="gpt-4o"),
    db=SqliteDb(db_file="tmp/agent.db"),
    session_id="my_session",
    cache_session=True,  # Enable in-memory caching
)

# First run loads from database and caches
agent.run("First message")

# Subsequent runs use cached session (faster)
agent.run("Second message")
```

## Session Management Flow

Here's how sessions are managed across runs:

```
Run 1: Create Session
├─ Generate/use session_id
├─ Initialize session in database
└─ Return session_id

Run 2: Continue Session
├─ Provide same session_id
├─ Load session from database/cache
├─ Add new run to session
└─ Save updated session

Anytime: Manage Session
├─ Set/get session name
├─ Retrieve session state
└─ Access session metrics
```

## Performance Considerations

| Feature | Use Case | Performance Impact |
|---------|----------|-------------------|
| Custom session IDs | System integration | No impact |
| Session names | UI/UX improvement | Minimal (one-time generation) |
| Auto-generated names | Automatic labeling | Small (requires model call) |
| Session caching | High-volume systems | Significant improvement (avoids DB reads) |
